# IYPT-Skill

一套 Claude Code skill，让 Claude 按流水线完整攻下一道 IYPT（国际青年物理学家锦标赛）题目：

```
物理分析 ⇄ 对抗式审稿  →  仿真/可视化 ⇄ 美学审查  →  PPT ⇄ 美学审查
```

## 它解决什么问题

IYPT 的题目是**故意欠定的**——"Investigate the phenomenon and explain the dependence on relevant parameters"，不给几何、不给材料、不给参数范围。直接让模型"解这道题"，产出的东西通常有三种病：

1. **隐式假设**——用了"均匀磁化""准静态""小角近似"却从没说出口，也没说什么时候它会崩；
2. **口头忽略**——"空气阻力可忽略"，但没给任何数字证明它可忽略；
3. **脆弱结论**——结论其实高度依赖某个没被检验的简化，一旦对手（Opponent）问到就塌。

所以 `iypt-analysis` 的核心不是"解题"，而是**把欠定的题目补全成一个可证伪的物理问题**：先写「设定书」把题目没给的条件显式定死，再用「假设台账」把每条简化连同它的**不等式成立判据**和失效边界一起记账，再用「机制预算」强制每个"忽略"都必须给出数量级数字。写完之后，一个 **fresh-context 的对抗式审稿人**拿着一份具体的物理错误模式清单反复攻击它——不是泛泛地"检查一下对不对"。

## 安装

### 方式一：作为 plugin（推荐）

```
/plugin marketplace add Alumin-Hydro/IYPT-Skill
/plugin install iypt@iypt-skill
```

### 方式二：直接拷贝 skills 目录

```bash
git clone https://github.com/Alumin-Hydro/IYPT-Skill.git
cp -r IYPT-Skill/skills/* ~/.claude/skills/
```

两种方式共用同一份 `skills/` 文件。

## 用法

把题目原文丢给 Claude，让它跑分析：

```
用 iypt-analysis 分析这道题：
"A cylindrical magnet is dropped into a vertical conducting pipe. Investigate the
motion of the magnet and how the terminal velocity depends on relevant parameters."
```

它会在 `iypt/<problem-slug>/` 下产出：

| 文件 | 内容 |
|---|---|
| `00-problem.md` | 原题 + 题型判定 + **设定书**（把题目没给的条件全部定死） |
| `01-analysis.md` | **主交付物**：假设台账、量纲分析、机制预算、分层推导、可证伪预测、实验方案 |
| `01-review-r{n}.md` | 每一轮的对抗式审稿报告 |
| `handoff/model-spec.json` | 交给下游仿真的机器可读契约 |

然后跑仿真：

```
用 iypt-simulation 把 iypt/magnetic-brake 的模型算出来
```

| 文件 | 内容 |
|---|---|
| `02-sim/acceptance.md` | **★ 验收断言**：每条 `expected_shape` 的机械可判翻译（**写代码之前先写它**） |
| `02-sim/code/` | Python 源码（一键复现）+ `matlab/`（带自检的移植） |
| `02-sim/figures/` | 静态图 PNG + SVG，每张都盖 `SIMULATION` 戳 |
| `02-sim/interactive/` | 自包含单文件 HTML 动画（无 CDN，可离线放、可塞进 PPT） |
| `02-sim/results.json` | **交给 Skill 3 的机器可读契约**：每道门、每条断言的判定 |
| `02-sim/model-challenge-r{n}.md` | 数值打脸模型时的**举证包**（仅在触发反向边时产出） |

分析写完后，Skill 1 会**自动**派一个 fresh-context 的审稿人来攻击它，按洞修订，最多 3 轮。3 轮还闭不上的窟窿，它会**诚实地在文首标注 `[GAP]`，而不是假装通过**。

单独复审一份已有的分析：

```
用 iypt-physics-review 审一下 iypt/magnetic-brake/01-analysis.md
```

## `iypt-simulation` 的核心不是"画图"，是"证伪"

给一个模型让 AI 写仿真代码，产出的东西通常有**三种病**：

1. **图是"画"出来的，不是"算"出来的** —— 结果先跑出来，然后看着图说"嗯挺合理的"。**你看着任何一条曲线都能编出一套解释。**
2. **代码错和模型错分不开** —— 曲线不对，是积分写错了还是物理想错了？分不开就只好两头猜；两头猜不出来，**最后改的一定是图**。
3. **"验证"退化成"自我确认"** —— 为了写代码，你必然把物理重推一遍。然后你会拿数值结果跟**自己心里的预期**比，而不是跟 Skill 1 白纸黑字承诺的东西比。

所以：

- **`figures[].expected_shape` 是 Skill 1 在开算之前就写死的曲线形状**——这是**预注册**（pre-registration），和临床试验先注册终点再收数据是同一个道理。
- **Skill 2 的第一个动作不是写代码，是写验收断言**（`02-sim/acceptance.md`），把每条 `expected_shape` 翻译成机械可判的数值断言。**顺序颠倒，断言就会退化成"数值结果的复述"。**
- **每条断言必须逐字抄写契约原文**（`quoted_expectation`）。`check_sim.py` 会机械校验它确实是 `model-spec.json` 对应字段的**子串**——抄不上就是 ERROR。这是把「零号规则」从劝诫变成检查。
- **Gate 0（极限对拍）先跑**：数值解在某个极限下必须回到闭式解。它是**纯数学恒等式，与物理对错无关**——于是它把"代码对不对"和"物理对不对"**解耦**了。没有它，后面每个不符都是糊涂账。
- **数值打脸模型时，走反向边**（`docs/pipeline.md` §5）：Skill 2 无权改模型、更不许把图改好看，它产出一份带举证的 `model-challenge-r{n}.md` 回送 Skill 1，并带四条**防捏合铁律**（单向棘轮：修订只能让预测**变弱**）。
- **仿真必须标注为仿真**：每张图盖 `SIMULATION` 戳，且 `check_sim.py` 直接在 SVG 里 grep 它——**这个检查伪造不了。**

## 样例

`examples/magnetic-brake/` 是一次真实跑通的完整产出（"圆柱磁体在导电铜管中下落，研究终速对参数的依赖"），可以直接当作参考基线：设定书、6 条假设台账、Buckingham Π、机制预算、闭式解 $v_t = \frac{1024}{45}\frac{Mga^4}{\mu_0^2m^2\sigma w} = 2.74$ cm/s、7 条零自由参数预测、实验方案，以及审稿报告。

这个样例最值得看的地方是**它的模型是"知道自己错在哪"的**：点偶极子近似要求 $a \gg L$，而实际 $a/L = 0.60$——判据根本不满足。台账把它标为 `RISKY`，并明确指出：这**不影响** $v_t \propto \sigma^{-1}, w^{-1}, B_r^{-2}$ 的指数，但**会让最激进的预测 $v_t \propto a^4$ 站不住**。这种"我的结论哪一条会先塌"的自觉，正是 Physics Fight 上分数的来源。

### 审稿人真的会咬人吗

做了两组对照测试：

- **对干净的分析**：审稿人独立重推了整条链条（含 Beta 函数积分、量纲代数），数值逐位吻合，判 `MINOR`——但**仍抓出 3 处真实缺陷**：符号 $R$ 既是磁体半径又被用作圆环电阻（P13）；"达到终速距离"用了 $v_t\tau\times4.6$ 这个错误捷径，系统性高估 27%（精确值 0.28 mm）；扫描端点检查只覆盖了空气阻力，漏了自感与趋肤。**三处都已按报告修订**。
- **对故意注入 3 个错误的版本**：判 `MAJOR`，抓到了量纲错（并证明单位会变成 1/s、数值会变成比自由落体还快的 4.57 m/s）和口头忽略。

但它**漏掉了第三个注入错误**：文档里白纸黑字写着 $dE/dt = +bv^2 > 0$（系统自发增能，公然违反热二），审稿人却判"P10 未命中"——因为它自己重推了一遍能量方程、推对了，就默认作者写的也对。**它验证的是自己的推导，不是作者写下的字。** 这条教训已经作为「零号规则」写进 `physics-failure-patterns.md` 的最前面。

### 那仿真的门会不会也是摆设

同样做了注入测试（`examples/magnetic-brake/02-sim/code/smoke_test.py`）——**四个注入，四道不同的门，全部抓到，基线不误报**。

| 注入 | 抓它的门 |
|---|---|
| 漏掉一个 $\mu_0$（量纲错） | Gate 0 极限对拍（误差 $4.5\times10^{13}$%） |
| **点偶极子场冒充有限长磁体的场** | **AS-27 结构型 `must_not`** |
| 广义积分截断从 $200a$ 砍到 $1.2a$ | Gate 0（误差 8.46%） |
| **截断长度写成绝对值（扫描端点悄悄失效）** | **Gate 1 的扫描端点收敛检查** |

**其中两条是被冒烟测试逼出来的，不是设计时想到的**——这才是它的价值：

- **`must_not` 不能只检查"拟合出来的数"，必须检查"结构"。** 第一版只有"斜率若回到 4.00 则代码错"。注入「点偶极子场 + 正确的径向积分」（**只少一个修正**）后，斜率落在 **3.79**——既不是陷阱值 4.00，也不是真值 3.44，**从两条断言之间溜过去了**。修法：查**涡流峰位**——点偶极子的峰解析地落在 $a/2$，有限长磁体的落在 $L/2$，**那是场的结构性质，不是拟合量**。注入之后峰位精确落在 $a/2$，一击即中。
- **收敛门必须在扫描端点上做，不能只在基准点做。** 把广义积分的截断写成绝对值，基准点上一切正常、Gate 0 也过（0.017%），但大 $a$ 端每个点都偏小一点——**拟合出的斜率是 3.4509，真值 3.4392，差 0.012，肉眼完全看不出来**。这和 Skill 1 的「机制预算必须做扫描端点检查」是同一个道理。

## 一个诚实的说明

真实 IYPT 的评分中**实验占很大权重**，且必须真正动手做。这套流水线用**数值仿真**替代实验，两者不是等价物。因此：

- `01-analysis.md` 里仍然会写出**可执行的实验方案**（器材、测量方法、误差来源），供你真的去做；
- 仿真结果在 PPT 中必须**明确标注为仿真**，绝不伪装成实验数据。
