# IYPT 数值仿真与验证

## 这个 skill 到底在解决什么

给一个模型让 AI 写仿真代码，产出的东西通常有**三种病**：

1. **图是"画"出来的，不是"算"出来的。** 结果先跑出来，然后看着图说"嗯，挺合理的"。——**你看着任何一条曲线都能编出一套解释。** 事后的合理化没有任何证据价值。
2. **代码错和模型错分不开。** 曲线不对，到底是积分写错了，还是物理想错了？分不开，就只好两头猜；两头猜不出来，最后改的一定是图。
3. **"验证"退化成"自我确认"。** 为了写代码，你必然把物理重推一遍。然后你会拿数值结果跟**自己心里的预期**比对，而不是跟 Skill 1 白纸黑字承诺的东西比对。**你验证的是你自己，不是那份契约。**

所以：**这个 skill 的核心不是"画图"，是"证伪"。**

`figures[].expected_shape` 是 Skill 1 **在开算之前**就写死的曲线形状。它同时是任务书和验收标准。这是**预注册**（pre-registration）——和临床试验先注册终点、再收数据，是同一个道理，也是同一个理由。

## 五件真正改变结果的事

1. **先写验收断言，再写求解器。** Stage 1 不许碰任何求解代码。先把每条 `expected_shape` 翻译成机械可判的数值断言，落成 `acceptance.md`。**顺序颠倒，断言就会被写成"数值结果的复述"。**
2. **逐字抄写被验对象的原文。** 每条断言必须带 `quoted_expectation`——**从 `model-spec.json` 里一字不改地抄过来**。抄写这个动作本身就是强制你去读它。`check_sim.py` 会机械校验这段引文确实是原文的子串。
3. **Gate 0（极限对拍）先跑，跑不过不许往下走。** 它是纯数学恒等式，与物理对错无关。**它把"代码对不对"和"物理对不对"解耦**——这是整个验证阶梯的地基。
4. **"符得太好"也是失败——而且 `must_not` 必须查「结构」，不能只查「拟合出来的数」。** 如果完整模型算出的斜率恰好回到玩具模型的解析值，那不是"验证成功"，那是**修正项根本没进代码**。
   > 但**光查斜率不够**：真实的 bug 往往**只少一个修正**，斜率会落在"陷阱值"和"真值"**之间**，从两条断言中间溜走。**必须再加一条直接检查解本身的结构性断言**（峰位、节点数、对称性——那些在正确/退化模型下**离散地不同**的量）。这一条是实测被 bug 溜掉之后才补上的，见 `references/verification-ladder.md`。
5. **收敛门要在扫描端点上做，不能只在基准点做。** 和 Skill 1 的「机制预算必须做扫描端点检查」同一个道理。
6. **仿真必须标注为仿真。** 每张图盖 `SIMULATION` 戳。这是底线，不是风格问题。

## 允许联网，但有一条红线

允许 WebSearch / WebFetch 查**数值方法**和**库 API**（"scipy 有没有完全第三类椭圆积分"、"solve_ivp 怎么处理刚性"）。

**红线：不许去查"答案应该是多少"。**

查到一篇论文说"磁体在铜管里的终速约 3 cm/s"，然后回来调你的代码去凑这个数——**这是拟合，不是计算**，而且是在代码层面犯审稿模式 P12。你的数字必须完全来自 `model-spec.json` 里的参数和方程，不能来自任何"我知道答案大概长什么样"。

**你可以查怎么用 `ellipk`，不许查结果应该是几。**

# 工作流

工作区 `iypt/<problem-slug>/`，产物路径见 `docs/pipeline.md` §2。

## Stage 0 · 读契约，不读散文

**只读 `handoff/model-spec.json`。不要打开 `01-analysis.md`。**

这不是官僚主义。契约的意义就在于它是**充分**的——如果 JSON 里的信息不够你算出结果，那是 Skill 1 的缺陷，你应该报 `SPEC-DEFECT`（见 Stage 9），而不是跑到散文里去找补。**去散文里找补，等于替 Skill 1 掩盖了一个契约漏洞，下一道题它还会犯。**

读完打印一份清单，确认你知道要干什么：

```
equations:              N 条（其中 model-0 / model-1 / model-2 各几条）
targets:                M 个（几个有 analytical_prediction？）
figures:                K 张（几张是 animation/interactive？）
risky_assumption_checks: J 条（必须全跑）
极限对拍配方:            equations[].numerical_notes 里有吗？没有 -> SPEC-DEFECT
```

## Stage 1 · 先写验收断言（`02-sim/acceptance.md`）

> **这一步不许写任何求解器代码。**

把每条自然语言翻译成机械可判的数值断言。来源有四类：

| 来源 | 翻译成 |
|---|---|
| `figures[].expected_shape` | 曲线断言（斜率 / 单调性 / 渐近值 / 坍缩散布 / 峰位） |
| `targets[].analytical_prediction` | 对拍断言（数值解 vs 闭式解的相对偏差） |
| `equations[].numerical_notes` 里的极限对拍 | **Gate 0 断言**（最优先） |
| `risky_assumption_checks[].pass_criterion` | RISKY 验证断言 |

每条断言写全六项（模板见 `templates/acceptance-template.md`）：

1. **`quoted_expectation` —— 逐字抄写原文。** 一个字都不许改。
2. **`interpretation` —— 翻译时的判读。** `expected_shape` 是自然语言，**有歧义时必须写清你选了哪个读法、为什么**。
3. **`assert_kind`** —— `slope` / `value` / `monotonic` / `asymptote` / `collapse` / `peak` / `deviation` / `limit` / `must_not`
4. **`expect`** —— 机械可判的表达式，如 `slope == -1.00`、`|k - 4.0| > 0.3`、`scatter < 1%`
5. **`tolerance`** —— 容差，**以及它是怎么定出来的**
6. **`verdict_if_violated`** —— 违反时判 `FAIL-CODE` 还是 `FAIL-MODEL`（规则见下）

### 判定规则（写代码之前就定死）

| 被违反的断言 | 判定 | 为什么 |
|---|---|---|
| `limit`（Gate 0 极限对拍） | **FAIL-CODE** | 纯数学恒等式，与物理无关。对不上必是代码错。 |
| `must_not`（"符得太好"） | **FAIL-CODE** | 修正项根本没进代码。 |
| 收敛门 | **FAIL-CODE** | 网格 / 容差 / 截断污染了结果。 |
| Skill 1 承诺的定性形状（**Gate 0 已过的前提下**） | **FAIL-MODEL** | 代码已被证明是对的 → 物理错了 → 反向边。 |

### `expected_shape` 是**双向**的

**"不符"是问题，"符得太好"也是问题。** 看这条真实的 `expected_shape`：

> "Model-0 是斜率恰为 +4.00 的直线。**Model-2 预期斜率显著小于 4**。**若 Model-2 也给出 4.00，说明有限尺寸修正没被正确实现——回去查代码**"

Skill 1 预先埋了一个陷阱：Model-2 若算出 4.00，那是失败。**每条 `expected_shape` 都要问一遍：有没有一个"太漂亮以至于可疑"的结果？有的话，把它写成 `must_not`。**

### 容差不许事后放宽

**放宽容差是"把图改好看"的另一种写法**，而且更隐蔽。容差必须在 Stage 1 定死并写清理由。跑完之后发现差一点点过不了 → **那就是没过**。

## Stage 2 · Gate 0：极限对拍

> **先跑它。跑不过，后面一律不许走。**

`equations[].numerical_notes` 里通常写好了配方："数值解在某某极限下必须回到闭式解，误差 < X"。

**照做，但先在纸上验一遍这个配方本身对不对。**

> **血泪教训（magnetic-brake，真实发生）**：配方原文是"令 $L\to0$ 且保持 $m = M_s V$ 固定（退化为点偶极子）、同时令 $w\to0$"。照做，比值收敛到 **3.55，不是 1**。
>
> 原因：只让 $L\to0$ 而保持 $R = 5$ mm，得到的是一个**半径 5 mm 的薄圆盘**——在 $r = a = 6$ mm 处，它根本不是点偶极子（$k^2 = 4Ra/(R+a)^2 = 0.99$，离远场十万八千里）。点偶极子极限要求**整个磁体**相对 $a$ 缩小：$R$ 和 $L$ 必须**一起**趋于 0。
>
> **它收敛到一个稳定值、但不是 1——这是"极限存在，但取错了极限"的典型指纹。** 认得出这个指纹，你就知道该报 `SPEC-DEFECT` 而不是去改代码。

**Gate 0 的极限一定要扫一串**（如 $\epsilon = 1, 0.3, 0.1, 0.03, 0.01$），看误差是否**单调趋于 0**。只取一个点，你分不清"通过了"和"碰巧"。

## Stage 3 · 分层实现：Model-0 → Model-1 → Model-2

**上来就写完整数值模型，你会调三天参也不知道哪错了。**

- **Model-0**：`closed_form` 直接代入。**它必须精确重现 `targets[].baseline_value`**——重现不了，是你参数读错了或单位错了，先修这个。
- **Model-1**：修正项。
- **Model-2**：完整数值。

**每一层算完，立刻和上一层在重叠区对拍**（Gate 2）。

**参数一律从 `model-spec.json` 载入，不许硬编码。** 硬编码的数字会和契约悄悄漂移，而且 `check_sim.py` 查不出来。

## Stage 4 · Gate 1：收敛门

**结果必须与数值参数无关。** 三件事各做一遍：

- 容差收紧 10×（`rtol: 1e-8 → 1e-9`）
- 求积网格加密 2×
- 广义积分的截断范围扩大 2×

**结果变化必须 < 断言容差的 1/10。**

> **不做这个，你那个"斜率 -1.02"可能纯粹是网格太粗。** 然后你会以为发现了物理。

## Stage 5 · Gate 2 / Gate 3：分层对拍 + 解析对拍

每个 `targets[].analytical_prediction` 都要算出数值解和解析预测的**相对偏差**，写进 `results.json`。

**偏差大不一定是错。** Model-2 和 Model-0 的差距，正是某条 RISKY 假设崩溃的**量度**——那可能恰恰是本题最有价值的结果。但必须**记录**，并在 `verdict_note` 里解释清楚这个偏差来自哪条假设。

## Stage 6 · Gate 4：跑全部断言

`acceptance.py` 全跑，逐条给 `measured` 和 `verdict`。

**不许有"未评估"的断言。** 跳过一条 = 契约违约。

## Stage 7 · Gate 5：RISKY 验证任务

`risky_assumption_checks[]` **每条都必须跑**，按它自己的 `pass_criterion` 判定。

**假设没站住不是灾难。** 标出来的 RISKY + 一个验过它的数值实验 = 满分答案；藏起来的 RISKY = 被 Opponent 一击致命。

**注意 `pass_criterion` 里可能预先注册了应对动作**：

> "若 $|k-4| > 0.3$，则 P5 被数值判定为不可靠，**必须在 `01-analysis.md` 的预测表中把 P5 降级并说明**。"

触发了就**照办**，标 `PRESCRIBED`。**这不是事后合理化**——动作是在数据出现之前写下的。（见 Stage 9 和 `docs/pipeline.md` §5.4。）

## Stage 8 · 出图 + 交互页面 + MATLAB 移植

读 `references/figure-style.md`。要点：

- **每张图盖 `SIMULATION` 戳。** 通过 `figkit.py` 的统一入口出图，`check_sim.py` 机械检查。
- **断言结果直接画在图上**：expected vs measured。**读者不该需要翻 JSON 才知道这张图过没过。**
- **loglog 图必须标注拟合出的斜率**——别让观众自己拿尺子量。
- 轴标签必须带单位（与 `symbols[].unit` 对齐）。
- Physics Fight 是**投影 + 打印**：大字号、高对比、灰度可辨。
- `kind: animation` / `interactive` → 自包含单文件 HTML（无 CDN），**数据从 Python 结果烘焙进去**，页面不能自己算出一套和 `results.json` 不一致的数。

**MATLAB 移植**：仅在 Python 全部门通过后生成。文件头必须写明"**生成时未在 MATLAB 中执行**"，并附 `verify.m`——读已验证的 `results.json`、重算、逐项打印 PASS/FAIL。`results.json` 里 `matlab_port.verified` **必须是 `false`**。

> 谎报 `verified: true` 违反这个 repo 的立身之本。**未经执行的代码不是"验证过的代码"。**

## Stage 9 · 写 `results.json` + 机械检查 + 判定 status

```bash
python <skill_dir>/scripts/check_sim.py iypt/<problem-slug>
```

**ERROR 必须清零。**

然后按 `docs/pipeline.md` §5 判 status：

| status | 什么情况 | 怎么办 |
|---|---|---|
| `PASS` | 全部断言通过 | 交给 Skill 3 |
| `FAIL-CODE` | Gate 0 / `must_not` / 收敛门 不过 | **不许交付。** 改代码，重来。 |
| `SPEC-DEFECT` | 契约本身有毛病 | 修契约，重跑。**不需要重跑物理审稿**。 |
| `PRESCRIBED-REVISION` | 命中 Skill 1 预先注册的分支 | 照它写的动作执行。**不走审稿循环**。 |
| `MODEL-CHALLENGED` | Gate 0 已过，但 Skill 1 承诺的断言被违反，且无预注册应对 | 走反向边 ↓ |

### `SPEC-DEFECT` 有一条硬门槛

> **如果你必须引用仿真数字才能说明"契约写错了"，那它就不是 `SPEC-DEFECT`——它是模型被数据打脸了。**

`SPEC-DEFECT` 要求你能**在不看任何仿真结果的情况下**证明契约有毛病（文档自相矛盾、配方在数学上不可能成立、参数缺单位）。

**这条线必须画死，否则 `SPEC-DEFECT` 就是一个逃生舱**：任何一个不想面对"我的模型错了"的 agent，都可以把它重新描述成"哦，`expected_shape` 只是措辞不当，我改改措辞"，然后从便宜的门溜走。

### 反向边（`MODEL-CHALLENGED`）

1. 归档 `handoff/model-spec-r{n}.json`（**不覆盖**）
2. 写证据包 `02-sim/model-challenge-r{n}.md`：
   - 哪条断言被违反，Skill 1 **逐字**承诺了什么，实测是什么
   - **Gate 0 的通过记录** —— 这是"不是我代码错"的举证责任，必须给
   - 收敛门记录 —— 排除"网格太粗"
   - 一条命令的复现脚本
3. 派 agent 调 `iypt-analysis` 修订，prompt 里**必须**塞进这四条铁律：

```
你在修订一份被数值结果打脸的分析。铁律：

1. 修订必须指向一个**物理理由**（漏了一项 / 符号错 / 边界条件错 / 某条假设
   崩得比预想更厉害）。"为了和数值吻合"不是理由。
2. **单向棘轮**：修订只能让预测**变弱**（加适用条件、缩小适用域、降级），
   不能凭空变强。
3. 新的 expected_shape **必须能从修订后的模型独立推导出来**，推导写进
   01-analysis.md。**若新的 expected_shape 恰好就等于数值结果，那它不是
   预测，是事后拟合（P16）。**
4. 修订后重跑 check_analysis.py，并重过 iypt-physics-review 审稿。审稿 prompt
   里要写明："这次修订是被数值结果逼出来的，请重点查 P16 事后合理化。"
```

4. **最多 2 轮。** 超限 → `01-analysis.md` 文首 `[GAP]` + `open_gaps[]` + `status: GAP`，诚实弃权。

## Stage 10 · 注入式冒烟测试（做完之后，必须做这个）

**你怎么知道你那些门不是摆设？** 往代码里**故意注入 bug**，看阶梯抓不抓得到。

Skill 1 当初就是靠这个才发现「零号规则」的。Skill 2 的对应动作是往完整模型里注入：

| 注入 | 应该被哪道门抓到 |
|---|---|
| 漏掉一个物理常数（量纲错） | Gate 0（极限对拍会差数量级） |
| **偷偷用玩具模型的解冒充完整模型** | **`must_not`**（Gate 0 天然抓不到这个！） |
| 广义积分截断太短 / 网格太粗 | Gate 0 或 Gate 1 |
| **截断长度写成绝对值而非自然尺度的倍数** | **Gate 1 的扫描端点检查** |

**任何一个没被抓到，那道门就是摆设，必须重新设计。**

**顺带必须验证：基线（无注入）不被任何门抓到。** 会误报的门比没有门更糟。

> **实测：magnetic-brake 的 4 个注入里，有 2 个是靠冒烟测试才发现漏掉的**——「结构型 `must_not`」和「扫描端点收敛检查」两条，都是设计时没想到、被注入的 bug 打出来的。**冒烟测试不是走过场，它是这套阶梯唯一可信的自证。**

## 完成的标准

不是"图很好看"，是这七件事都做到了：

1. **`acceptance.md` 是在写求解器之前写的**，每条断言都带逐字引文
2. **Gate 0 过了**，而且极限扫了一串、误差单调趋于 0
3. **收敛门过了** —— 结果与网格 / 容差 / 截断无关，**且在扫描端点上也成立**
4. **每条 RISKY 假设都被数值验过**，预注册的应对动作都执行了
5. **每张图都盖了 SIMULATION 戳**，且断言结果标在图上
6. **一条命令能重现全部数字和图**
7. **做过注入式冒烟测试**，每道门都真的抓到了它该抓的东西

## 常见的自我背叛

- **先跑出结果，再写 `acceptance.md`** —— 那不是断言，是数值结果的复述。**顺序就是全部。**
- **`quoted_expectation` 是"复述"而不是"抄写"** —— 一复述就走样，走样的方向永远是对自己有利。逐字抄。
- **曲线不符，于是把容差从 5% 放宽到 20%** —— 这就是把图改好看，只是换了个地方改。
- **Gate 0 只取一个 $\epsilon$，误差 0.08%，判"通过"** —— 你不知道它是收敛了还是碰巧。扫一串。
- **Model-2 算出的斜率恰好等于 Model-0 的解析值，于是很高兴** —— **这大概率说明修正项没进代码。** 高兴错了。
- **`must_not` 只写了"斜率别等于陷阱值"** —— **不够。** 只少一个修正的 bug，斜率会落在陷阱值和真值**之间**，从两条断言中间溜走（实测：3.79，而陷阱在 4.00、真值在 3.44）。**必须再加一条结构性断言。**
- **收敛门只在基准点跑** —— 绝对截断长度会在扫描的一端悄悄失效。基准点上一切正常，Gate 0 也过，**而斜率已经错了**（实测：3.4509 vs 真值 3.4392，肉眼看不出来）。
- **把"我的模型错了"重新描述成"契约措辞不当"** —— 这是 `SPEC-DEFECT` 逃生舱。需要引用数字才能说明的，一律不是 SPEC-DEFECT。
