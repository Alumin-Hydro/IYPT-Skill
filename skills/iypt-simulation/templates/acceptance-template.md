# 验收断言 · `<problem-slug>`

> **这份文件必须在写任何求解器代码之前完成。**
>
> 顺序颠倒，断言就会退化成"数值结果的复述"——而你不会察觉，因为它读起来完全合理。
>
> 生成于 `handoff/model-spec.json`（版本 `r<n>`）。

## 怎么读这份表

每条断言六个字段：

| 字段 | 规矩 |
|---|---|
| **引文** | **逐字抄写** `model-spec.json` 的原文。一个字都不许改。`check_sim.py` 会验证它确实是原文的子串。 |
| **判读** | `expected_shape` 是自然语言，会有歧义甚至自相矛盾。**你选了哪个读法、为什么** —— 写在这里，交给人眼审。 |
| **断言** | 机械可判的表达式。`slope == -1.00`、`\|k - 4.0\| > 0.3`、`scatter < 1%` |
| **容差** | 以及**它是怎么定出来的**。**不许事后放宽**——放宽容差是"把图改好看"的另一种写法，只是更隐蔽。 |
| **违反时** | `FAIL-CODE` 还是 `FAIL-MODEL`（规则见下） |
| **实测 / 判定** | 跑完之后回填。**不许留空。** |

### 违反时的判定规则（写代码之前就定死）

| 被违反的断言 | 判定 | 为什么 |
|---|---|---|
| `limit`（Gate 0 极限对拍） | **FAIL-CODE** | 纯数学恒等式，与物理无关。对不上必是代码错。 |
| `must_not`（"符得太好"） | **FAIL-CODE** | 修正项根本没进代码。 |
| 收敛门 | **FAIL-CODE** | 网格 / 容差 / 截断污染了结果。 |
| Skill 1 承诺的定性形状（**Gate 0 已过的前提下**） | **FAIL-MODEL** | 代码已被证明对 → 物理错了 → 反向边。 |
| 命中 `pass_criterion` 里**预先注册**的分支 | **PRESCRIBED** | 不是失败。照它写的动作执行。 |

### `must_not`：**"符得太好"也是失败**

**每条 `expected_shape` 都要问一遍：有没有一个"漂亮得可疑"的结果？**

典型的：完整数值模型算出的斜率，**恰好**等于玩具模型的解析值。那不是"验证成功"——**那是修正项根本没进代码，你只是把 Model-0 换了个壳又跑了一遍。**

有这种可能，就写一条 `must_not` 断言，违反时判 `FAIL-CODE`。

## 收敛门（Gate 1）

**结果必须与数值参数无关。** 三件事各做一遍：

| 拧什么 | 拧多少 | 结果变化 | 判定 |
|---|---|---|---|
| 容差 | 收紧 10× | _(回填)_ | |
| 求积网格 | 加密 2× | _(回填)_ | |
| 广义积分截断范围 | 扩大 2× | _(回填)_ | |

**变化必须 < 断言容差的 1/10。** 不做这个，你那个"斜率 −1.02"可能纯粹是网格太粗——然后你会以为自己发现了物理。

## 人眼审这份表时该问的三个问题

1. **每条引文，真的是抄的吗？** —— 复述一次就走样，而走样的方向永远对自己有利。
2. **有没有哪条容差是"刚好能过"的？** —— 容差必须来自物理或数值精度，不是来自"这样就能过"。
3. **有歧义的 `expected_shape`，判读记录下来了吗？** —— 不记录，agent 会挑对自己有利的读法，**而且它自己都不会察觉**。
