# IYPT 设计审查

## 铁律 0：**你会脑补出一张比实际渲染出来的更好看的图**

这是这份 skill 里最重要的一句话。

你读过绘图代码，你知道每个 `annotate` 想指向哪里、每个 `legend` 想放在哪个角。**于是你在脑子里渲染出一张「代码的意图」，而不是「matplotlib 实际吐出来的像素」。** 然后你对着这张幻觉审稿。

**实际吐出来的东西经常是**：标注框跑到坐标轴外面、图例正好压住曲线的峰、对数轴的次刻度标签糊成一团、中文变成豆腐块 □□□ ——**而 matplotlib 一个警告都不会给你**。

> **所以：必须真的打开 PNG，用眼睛看。** 用 Read 工具读图片文件——你是多模态的，你**能**看。
>
> **读代码猜排版 = 没审。**

这条铁律和 `iypt-physics-review` 的「零号规则」同源：**凡是让模型检查一份它自己也理解的工作，它都倾向于验证「它以为的东西」，而不是「实际存在的东西」。**

---

## 这个 skill 在解决什么

IYPT 的图不是插图，**是论据**。而论据有三个物理约束：

1. **投影到会议厅大屏**，后排要读 → 大字号、高对比
2. **可能被打印成灰度**（评委手里的讲义）→ **色相会完全消失**
3. **观众里有色盲**（男性约 8%）→ 色相会部分消失

**结论：身份永远不能只靠颜色。**

但真正会毁掉一张图的，通常不是配色——**是标注把数据挡住了**。

> **实测（真实发生）**：F-4 是数据坍缩图，全篇杀伤力最大的一张。它的信息是"Model-0 的 5 组参数**精确坍缩到同一条水平线上**"。
>
> **而那个浮在轴内的断言框，正好盖住了那堆坍缩的点。**
>
> 图上每个元素都"有用"，配色也过了验证器。**但这张图的全部意义被它自己的注释挡住了——而作者没发现，因为他知道那些点在那儿。**

## 输入 / 输出

**输入**：`02-sim/figures/` 的 PNG + `02-sim/results.json`，或 `03-slides/` 的产物。

**输出**：`0X-design-review-r{n}.md`（`0X` = `02` 或 `03`，`n` = 轮次，**不覆盖上一轮**）。判定 `PASS` / `MINOR` / `REVISE`。

**只管美观度与可读性。** 物理正确性归 `iypt-physics-review`，数值正确性归 `iypt-simulation` 的验证阶梯。

**但有一个例外（D12）**：**图上的数字必须和 `results.json` 对得上。** 这条你要查——因为**没有别人查**。

# 流程

## 第 0 步 · 加载弹药

读 `references/design-failure-patterns.md`。**必须真的读**——那 14 条不是提纲，每条都带一个你要执行的**具体动作**，而且**每一条都是真实踩过的坑**。凭记忆做这件事会退化成"泛泛评价"，等于没审。

## 第 1 步 · **打开图，看**

```
Read  02-sim/figures/F-1.png
Read  02-sim/figures/F-2.png
…
```

**一张一张看。不要读绘图代码。**

> **代码告诉你作者想画什么；像素告诉你他实际画了什么。**

看每一张之前，**先读 `results.json` 里这张图的 `caption`**——它写的是"这张图**证明了什么**"。然后看图，然后问：

> **这张图想证明的那个东西，我第一眼看见了吗？还是被什么挡住了？**

## 第 2 步 · 逐条攻击（核心）

对 **D1 到 D14**，每一条都执行它的攻击动作。**不要跳过任何一条。**

记录三种结果之一：

- **命中**：写清楚**哪张图、哪个位置、为什么是问题、怎么修**
- **未命中**：一句话说明你做了什么检查
- **UNCLEAR**：看不出来（分辨率不够、需要原始数据）——**说清卡在哪，不要默认它没问题**

## 第 3 步 · 判定

| 判定 | 门槛 |
|---|---|
| **`REVISE`** | 命中的模式会**让读者看不见、或看错这张图要证明的东西**：数据被挡、豆腐块、刻度糊成一团、图上数字与 `results.json` 对不上、缺 SIMULATION 戳 |
| **`MINOR`** | 不影响信息传达（配色可以更好、间距不理想、标题措辞） |
| **`PASS`** | 14 条**逐条**看过，无 REVISE |

**`PASS` 的门槛是"我逐条看过并且没打穿"，不是"我扫了一眼觉得挺好"。**

> **缺 SIMULATION 戳一律判 `REVISE`。** 那不是排版问题——那是**把仿真伪装成实验数据**（`docs/pipeline.md` §7）。真实 IYPT 的评分里实验占很大权重，一张没标注的仿真图进了 PPT，在 Physics Fight 上会被当作学术不端。

## 第 4 步 · 写报告

写到 `0X-design-review-r{n}.md`：

```markdown
# 设计审查 · 第 {n} 轮

**判定**：REVISE | MINOR | PASS
**我打开并看过的图**：{逐个列出 —— **没打开的图不许判定**}

## 一句话结论
{最致命的那个问题是什么。如果 PASS：我逐条看了什么、为什么没打穿。}

## 命中的问题

### H1 · [REVISE] F-4 的断言框盖住了坍缩的那堆点 — 命中 D1
- **位置**：F-4，图正下方偏左，$\Pi_1 \approx 22.76$ 那条水平线上
- **为什么是问题**：这张图的**全部意义**就是"Model-0 的 5 个点精确堆到那条线上"。
  **框正好盖在那堆点上**，读者看不见这张图要证明的东西。
- **怎么修**：把断言条带移到坐标轴**外面**（`figkit.assertions()` 已经这么做了——
  检查这张图是不是绕过了 figkit）

### H2 · [MINOR] …

## 14 条模式的逐条结果
| 模式 | 结果 | 我做了什么检查 |
|---|---|---|
| D1 标注压住数据 | **命中 → H1** | 逐图看了 F-1…F-5；F-4 的框盖住了 22.756 线上的点 |
| D2 豆腐块 | 未命中 | 逐图确认无 □□□ |
| D3 对数轴刻度 | 未命中 | F-1(b) 的 w 轴显式设了刻度，未糊 |
| … | … | … |

## UNCLEAR
- {看不出来的，卡在哪，需要什么}
```

## 反向边：判定 `REVISE` 之后

报告交回 **Skill 2**（图）或 **Skill 3**（PPT），逐条修，然后**重新审**（`n` 递增，不覆盖上一轮）。

**最多 3 轮。** 3 轮之后仍有 REVISE → 在报告里诚实标注，不要假装通过。

**但有一条铁律**（和 `iypt-simulation` 共享）：

> **如果曲线与 `expected_shape` 矛盾，那不是设计问题，不许"把图改好看"。**
>
> 你的职权是"让读者看清楚数据"，**不是"让数据看起来更漂亮"**。发现有人在用配色/裁剪/换轴范围来掩盖一个数值上的矛盾 —— **那是 `iypt-simulation` 的 `FAIL-MODEL`，立刻指出来，不许放行。**

## 常见的失败 —— 你自己的失败

- **读了代码，没看图。** 见铁律 0。**这是这份 skill 唯一一种致命的失败方式。**
- **给了 PASS 但没跑完 14 条。** 最常见的自我背叛。跑完了再说。
- **只说"配色可以更协调些"。** **没用。** 要么指出**哪个元素挡住了哪个数据**，要么说清**哪个字后排看不见**。**定性评论不算审查。**
- **把 REVISE 说成 MINOR。** 问自己："**如果这张图就这样投在赛场大屏上，评委能看见它要证明的东西吗？**" 看不见 = REVISE。
- **没看 `results.json` 就判图上的数字。** D12 要求你对着契约核 —— 图上写着 "k = 3.44"，`results.json` 里是不是也是 3.44？
- **帮作者"把图改好看"来掩盖一个数值矛盾。** 见上面的铁律。**你不是化妆师。**
